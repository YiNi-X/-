# 后续更改计划

## 1. 文档目的

这份文档用于说明当前网站在后续阶段的数据接入与预测能力演进方案。

核心原则只有两条：

- 第一阶段优先保证演示稳定、结果可信、实现成本可控
- 第二阶段再逐步过渡到稳定 AIS 接入与准实时预测

当前网站定位仍然是论文成果的场景化演示平台，而不是已经完成生产级部署的港口业务系统。

---

## 2. 当前项目基础

目前代码依据中已经具备以下关键基础：

- AIS 点位数据
  - `代码依据/轨迹聚类/shaixuanhou_.csv`
  - `代码依据/轨迹聚类/cleaned_ais.CSV`
- 聚合后的流量矩阵
  - `代码依据/流量预测/grid_mmsi_count（流量数据）.csv`
- 已训练的流量预测模型权重
  - `代码依据/流量预测/save/model_0.pt`
- 预测相关代码
  - `代码依据/流量预测/复现STCGN.ipynb`
  - `代码依据/流量预测/load_data.py`
  - `代码依据/流量预测/stgcn.py`
  - `代码依据/流量预测/utils.py`

因此，网站后续升级不需要从零开始补建研究链路，而是可以直接站在现有数据、现有输入格式和现有模型权重之上继续推进。

---

## 3. 总体演进路线

### 第一版

- 不做实时后端
- 先离线生成预测输出文件
- 前端直接读取 `AIS 回放数据 + 预测结果数据`

### 第二版

- 如果以后拿到稳定 AIS，再做一个轻量后端
- 把新 AIS 映射到 `60` 个有效网格
- 再调模型滚动预测

---

## 4. 第一版详细方案

### 4.1 目标

第一版的目标不是“实时预测系统”，而是做出一个：

- 地图上船舶轨迹来自真实 AIS 历史数据
- 曲线、热区、数值变化来自已训练模型预测结果
- 前端观感接近实时，但整体实现仍然轻量稳定

### 4.2 数据组成

第一版建议把前端数据拆成两类：

#### A. AIS 回放数据

作用：

- 用于地图上的船舶点位、轨迹、船舶运动动画
- 提升网站真实性和说服力

建议来源：

- 优先使用 `cleaned_ais.CSV`
- 如果需要更原始的数据，可回溯 `shaixuanhou_.csv`

前端使用方式：

- 按时间片切分
- 每个时间片保留一批船舶位置、航向、速度、所属航路
- 前端只负责回放，不负责重新计算轨迹修复或聚类

#### B. 预测结果数据

作用：

- 用于左侧预测曲线
- 用于热区强度变化
- 用于右侧预警、协同建议、收益推演

建议来源：

- 通过 `grid_mmsi_count（流量数据）.csv` 作为输入
- 加载 `model_0.pt`
- 离线批量生成未来流量预测输出

前端使用方式：

- 读取已经导出的预测结果文件
- 不在浏览器里运行模型

### 4.3 第一版推荐文件形态

建议最终导出成两份前端可直读的数据文件：

#### `ais-playback.json`

建议包含：

- 时间片标识
- 每条船舶的 `id`
- 当前经纬度或相对画布坐标
- 航向
- 速度
- 所属航路编号

#### `flow-forecast.json`

建议包含：

- 时间片标识
- 当前总流量
- 下一窗口预测总流量
- 各热点网格当前值与预测值
- 各网格热度等级
- 模型输出对应的曲线点

### 4.4 第一版实际运行方式

前端页面运行时只做三件事：

1. 读取 AIS 回放数据
2. 读取预测结果数据
3. 根据当前时间片切换地图、热区、图表和文字说明

这样做的优点是：

- 不需要单独部署预测服务
- 不需要浏览器直接加载 PyTorch 模型
- 演示稳定
- 页面响应快
- 真实感明显提高

### 4.5 第一版开发任务拆分

#### 数据侧

- 从 `cleaned_ais.CSV` 提取网站回放所需字段
- 将 AIS 点位切分为网站可回放的时间片
- 根据现有 `grid_mmsi_count` 和 `model_0.pt` 离线生成预测结果
- 导出 `ais-playback.json`
- 导出 `flow-forecast.json`

#### 前端侧

- 新增 AIS 数据读取逻辑
- 用真实 AIS 数据替换当前部分示意船舶动画
- 新增预测结果文件读取逻辑
- 让热区、指标卡、预测曲线与导出结果联动

#### 文案侧

- 在页面说明中明确：
  - 船舶回放来自历史 AIS 数据
  - 流量预测来自已训练模型离线输出
  - 页面仍属于场景化演示版

### 4.6 第一版验收标准

- 地图中的船舶位置与轨迹不再完全依赖手工构造
- 热区与预测曲线不再只是手工填值
- 页面在无后端服务的情况下可独立运行
- 演示时可明确说明“展示层是真实 AIS 历史回放，预测层是模型输出结果”

---

## 5. 第二版详细方案

### 5.1 目标

第二版的目标是从“离线演示”升级到“准实时预测”。

这一版的重点不是重跑整条论文研究链路，而是让系统具备稳定接入新 AIS 并持续输出未来流量结果的能力。

### 5.2 第二版推荐架构

#### 在线部分

- 接入稳定 AIS 数据源
- 做基础清洗与时间片聚合
- 将新 AIS 映射到论文使用的 `60` 个有效网格
- 基于最近 `n_his` 个时间片构造模型输入
- 调用已训练模型做滚动预测
- 输出给前端展示

#### 离线部分

- 轨迹修复训练
- 主航路聚类与网格划分
- 邻接矩阵生成
- STGCN 模型训练与调优

也就是说，第二版在线运行的重点其实是：

`AIS -> 网格统计 -> 模型推理 -> 前端展示`

而不是每次都重新做：

`轨迹修复 -> 聚类 -> 预测`

### 5.3 第二版后端建议

推荐做一个轻量服务即可，例如：

- Python + FastAPI

建议提供这些接口：

#### `/api/current-ais`

返回：

- 当前时间片的船舶位置
- 当前地图回放所需字段

#### `/api/current-flow`

返回：

- 当前时间片 60 网格流量
- 当前总流量
- 当前热点网格列表

#### `/api/forecast`

返回：

- 下一窗口流量预测
- 各重点网格预测值
- 模型结果摘要

#### `/api/scene-summary`

返回：

- 当前系统状态
- 协同建议
- 预警摘要

### 5.4 第二版实时流程

建议采用如下流程：

1. 接收最新 AIS 数据
2. 做研究区域筛选
3. 将每条 AIS 点映射到对应网格
4. 统计每个时间片 60 网格的船舶数
5. 取最近一段历史窗口作为模型输入
6. 做标准化
7. 调用训练好的 STGCN 模型
8. 反标准化恢复真实流量值
9. 把结果推送给前端

### 5.5 第二版开发任务拆分

#### 数据侧

- 确认稳定 AIS 来源
- 定义时间粒度
- 统一 AIS 字段格式
- 建立 AIS 到 60 网格的映射规则

#### 模型侧

- 确认 `model_0.pt` 的加载方式
- 梳理 scaler、邻接矩阵、节点顺序
- 封装在线推理脚本

#### 后端侧

- 建立轻量 API
- 增加定时预测任务
- 输出前端可直接消费的数据结构

#### 前端侧

- 将当前页面从“静态时间片切换”升级为“准实时轮询或推送”
- 支持当前 AIS、当前流量、未来预测三层联动

### 5.6 第二版验收标准

- 前端能持续看到新 AIS 船舶变化
- 热区和曲线能跟随后端预测刷新
- 预测结果与论文模型输入格式一致
- 后端负载可控，页面可稳定演示

---

## 6. 两个版本的边界说明

### 第一版不做什么

- 不做实时后端
- 不做在线模型推理
- 不做全链路自动重跑
- 不做生产级 AIS 接入平台

### 第二版才开始做什么

- 稳定 AIS 接入
- 网格化在线统计
- 已训练模型在线推理
- 准实时前后端联动

---

## 7. 当前最推荐的落地顺序

如果后续工期有限，建议严格按下面顺序推进：

1. 先完成第一版
2. 用真实 AIS 历史回放替换现有部分演示船舶
3. 用模型离线输出替换现有部分手工预测数值
4. 页面讲通后，再考虑第二版轻量后端

原因是：

- 第一版已经足够明显提升真实性
- 第一版对现有网站改动小、风险低
- 第二版的真正难点在数据源稳定性，不在前端页面本身

---

## 8. 一句话总结

第一版的最佳方案是：

`历史 AIS 回放 + 模型离线预测输出 + 前端直接读取`

第二版的升级方向是：

`稳定 AIS 接入 + 60 网格映射 + 模型滚动预测`

这两步走法既符合当前网站“演示为主”的定位，也为后续升级到更强的系统版本保留了清晰路径。
